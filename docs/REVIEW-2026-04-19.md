## P1 — should fix before ship

### 1. `app/p/[id]/ResultPane.tsx:61,68` — clipboard errors swallowed

`navigator.clipboard.writeText()` can reject (insecure context, permission
denied, focus loss) but `handleCopyLink` and `handleCopy` flip `setCopied(true)`
regardless:

```ts
const handleCopy = async () => {
  await navigator.clipboard.writeText(content)  // can throw
  setCopied(true)                                // still fires
  // ...
}
```

User sees **"Copied!"** while nothing was copied to the clipboard.

**Fix**: wrap the `writeText` call in `try { ... } catch { … surface error or no-op …}`. Either bail before the `setCopied(true)` on reject, or show a short-lived "Copy failed" badge. Same fix in both `handleCopy` and `handleCopyLink`.

---

### 2. `app/dashboard/JobActions.tsx:24-30` — optimistic delete doesn't roll back

The delete handler removes the row from client state **before** the fetch,
then doesn't check `res.ok`:

```ts
if (onRemoved) onRemoved(pageUrl)  // row gone from UI
try {
  await fetch(`/api/p/request?pageUrl=${encodeURIComponent(pageUrl)}`, { method: "DELETE" })
  // no res.ok check
} finally {
  setBusy(false)
}
```

If the DELETE fails (network / 401 / 500), the row is gone from the
dashboard but still present in the DB — it pops back on the next page
reload. Confusing UX.

**Fix**: check `res.ok` and call an inverse `onRestore(pageUrl)` when the
parent supplies one; fall back to `router.refresh()` otherwise. Minor
API-surface expansion on the parent side but clear semantics.

## P2 — nice to have

### 3. `app/api/p/request/route.ts:23` — GET validates `pageUrl` lazily

Anon callers skip the `pageUrl` check:

```ts
export async function GET(req: NextRequest) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ inHistory: false })   // anon exits here

  const pageUrl = new URL(req.url).searchParams.get("pageUrl")
  if (!pageUrl) return NextResponse.json({ error: "pageUrl required" }, { status: 400 })
```

An anon caller with no `pageUrl` param gets `200 { inHistory: false }`; an
auth caller with no `pageUrl` gets `400`. Inconsistent. Safe either way
(anon always returns `false` regardless of URL) but slightly cleaner to
validate first.

**Fix**: move the `pageUrl` presence check above the auth branch.

### 4. `app/p/[id]/ResultPane.tsx:86` — POST `/api/p/request` has no rate limit

The add-to-dashboard endpoint is behind auth + same-origin but not
rate-limited. An authenticated user could spam to pollute their own
`user_requests` table. Blast radius is self-only (RLS-scoped, upsert
rather than insert, user can clean it up themselves), but still worth
a tiny bucket.

**Fix**: piggyback on the existing `SUBMIT` bucket (`consumeRateLimit(\`submit:user:${user.id}\`, rateLimit.AUTH_SUBMIT)`) or add a dedicated
tiny `add:*` bucket. Low effort; defence-in-depth.

---

### 5. `lib/upstash/rateLimit.ts:120-133` — fail-open with no health-check

When Upstash errors, rate limiting silently disables and the `console.warn`
routes to Sentry via `errorLog`. Correct. But there's no periodic health
check that Upstash is reachable, so a long outage could pass undetected
until someone hits the alerting thresholds in Sentry.

**Fix** (optional): a tiny `ping` every N monitor-cron ticks that calls
`redis.ping()` and logs an error on fail. Or rely on Upstash's own
uptime dashboard + configured Sentry alert on the `rateLimit.upstash`
context tag firing > N times in 1 h.

---

### 6. `app/p/[id]/ResultPane.tsx:40-57` — membership-check failure silent

The fetch is `fail-closed` (button hidden on network error), which is
correct, but there's no client-side Sentry capture for the error:

```ts
.catch(() => {
  if (!cancelled) setInHistory(true)   // hide the button, no log
})
```

If the GET `/api/p/request` starts failing in production for some
reason (e.g. the API route regresses), signed-in users silently lose
the save button.

**Fix**: `Sentry.captureException(err, { tags: { context: "ResultPane.membership" }})` in the catch block. Tiny change, non-blocking UX.

## Design-level notes (not bugs)

### `scrubError` — substring-based fallback is generic

`app/api/p/[id]/scrubError.ts` uses `lower.includes(...)` / `lower.startsWith(...)` mapping. Any pipeline error that doesn't match a known case falls through
to `"Couldn't generate a result for this site."`. That's safe (no leak) but
means new error categories are silently generic until added. Worth keeping in mind when adding new pipeline failure modes.

### `getPageByUrl` requires `result IS NOT NULL`

`lib/store.ts:getPageByUrl` filters for pages with a completed result. This
means `POST /api/p/request` with a URL that's still mid-crawl → 404. Good
invariant: user can only save finished pages.

(Not a gap: `ResultPane` only renders when the job is already `complete`/`partial`, so the save button can't appear before the result is written.)

### Vercel Analytics CSP

Confirmed during earlier deploy smoke tests that `@vercel/analytics` uses
the same-origin `/_vercel/insights/*` edge rewrite — no `connect-src`
addition was needed for `vercel.com` or `va.vercel-scripts.com`. Worth
re-checking if the SDK major-bumps.

### Homepage-rescue score floor (`score >= 15`)

`lib/crawler/enrich/group.ts` rescues the homepage into `optional` when
everything else filters out, gated on `score >= 15`. For jamlunch (chat
app with no headings and no meta) the score lands ~38-43 thanks to LLM
description + body-text weighting, so the rescue fires. For a genuinely
empty SPA the score would be ~0-10 and rescue skips — leaving the job
to fail cleanly via the "primary+optional both empty" path. Logic is
correct; worth re-testing if score weights in `score.ts` ever change.

### `pickInterval` "7 d" fallback

`lib/upstash/rateLimit.ts:pickInterval` falls through to `"7 d"` for
sub-daily rates. The `Ratelimit` interval type annotation is
`` `${number} ${"s" | "m" | "h" | "d"}` `` (no `"w"`), so `"7 d"`
is the correct max-window representation. For current configs we
never hit that branch — smallest rate is `12/3600 ≈ 0.003/sec`
(CRON_MONITOR) which resolves to `"1 h"` bucket of 12. Defensive
and correct.
